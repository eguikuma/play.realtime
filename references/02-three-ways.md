# 第 2 章 3 つのやり方 ポーリング SSE WebSocket

## この章で答える問い

第 1 章で、リアルタイムとは「画面を開きっぱなしにしておくと、サーバ側の出来事が勝手に届く」体験だと言葉にしました

これを実現するやり方は、歴史的に 3 つあります

ポーリング、SSE、WebSocket です

この章では、3 つそれぞれが「どんな会話の作法」なのかを、喩えと簡単な図で並べて見ていきます

実装の話はまだ出しません

「会話の形」だけを掴むのが目的です

## ポーリング —— 「もうある？」を繰り返し聞く

ポーリングは、ブラウザが自分から定期的にサーバに聞きに行くやり方です

数秒おきにリクエストを送って、「新しい何かありますか？」と確認します

```
ブラウザ ──「もうある？」── サーバ
ブラウザ ←─「ない」 ────── サーバ
ブラウザ ──「もうある？」── サーバ
ブラウザ ←─「ない」 ────── サーバ
ブラウザ ──「もうある？」── サーバ
ブラウザ ←─「これあるよ」── サーバ
```

しつこい子どもが「ねえ着いた？まだ？」と何度も聞いてくるイメージに近いです

仕組みとしては HTTP のリクエストを繰り返すだけなので、特別な技術は要りません

その代わりに 2 つの困りごとが出てきます

ひとつめは、ほとんどの問い合わせが空振りに終わるので、無駄なリクエストが多いこと

ふたつめは、確認の間隔ぶんだけ遅延が出ること

3 秒おきに聞きに行くと、ものごとが起きてから最大 3 秒は気付けません

## SSE —— サーバから流しっぱなしにしてもらう

SSE は Server-Sent Events の略です

ブラウザが最初に一度だけ「ここから何かあったら流してください」と声をかけると、サーバがその接続を開きっぱなしにして、出来事をそこに書き続けるやり方です

```
ブラウザ ──「ここから何か流してください」── サーバ
ブラウザ ←─「これあるよ」 ─────────────── サーバ
ブラウザ ←─「これもあるよ」 ───────────── サーバ
ブラウザ ←─「これも」 ─────────────────── サーバ
```

ラジオを 1 局つけっぱなしにしておくと、放送局の側で番組が始まるたびに勝手に流れてくる感じに近いです

聞き手のブラウザは黙って聞いているだけで、しゃべるのは送り手のサーバだけ、という片方向の通信です

ブラウザからサーバに何かを送りたいときは、別途 HTTP のリクエストを使います

```
ブラウザ ──「これ投稿します」── サーバ
```

その投稿は、開きっぱなしの SSE 経由で各ブラウザに流れていきます

```
ブラウザ ←─「投稿が来ました」── サーバ
```

「サーバ側の出来事を全員に届けたい」だけで足りる用途には、これで十分です

## WebSocket —— 同じ電話線で両方からしゃべる

WebSocket は、ブラウザとサーバの間に「両方向にしゃべれる通信路」を 1 本張るやり方です

最初に「これから WebSocket で話します」と握手のやり取りをしてからは、どちらからもメッセージを送れます

```
ブラウザ ──「招待を出します」 ── サーバ
ブラウザ ←─「相手が応答しました」── サーバ
ブラウザ ──「ミュートを切り替え」── サーバ
ブラウザ ←─「相手が切りました」 ── サーバ
```

電話に近いです

回線がつながっている間、お互いがいつでも話しかけられます

両方向に同じくらいの頻度でやり取りが必要な場面で、もっとも素直に書けます

ただし、両方向にメッセージが流せるぶん、サーバはクライアントから来るメッセージを受け取って、検証して、ルーティングする処理を持つことになります

片方向の SSE より、書く量も気を配る場所も増えます

## 3 つを並べて見ると

| やり方 | 接続の張り方 | しゃべる方向 | 向いている用途 |
| --- | --- | --- | --- |
| ポーリング | 短い接続を何度も張る | ブラウザが聞きに行く片方向 | 通信頻度が低く、特別な仕組みを入れたくない場面 |
| SSE | 1 本を張りっぱなしにする | サーバが流す片方向 | サーバ側の出来事を全員に届けたい場面 |
| WebSocket | 1 本を張りっぱなしにする | 双方向 | 両側から同じくらいの頻度でやり取りが要る場面 |

接続の張り方と、しゃべる方向が、3 つを分ける軸です

## どれを選ぶかの判断軸

学習者として身につけたい判断軸はひとつです

その機能で「サーバから一方向に流すだけで足りるか」を最初に問うこと

足りるなら SSE で十分です

足りないなら WebSocket を選びます

ポーリングは、HTTP の往復で済ませたい単純な確認や、SSE や WebSocket が使えない環境への保険として使う場面が多くなります

第 3 章で、この判断軸をりもどきの 4 機能に当てはめてみます
