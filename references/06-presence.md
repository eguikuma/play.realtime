# 第 6 章 「いる人」をどう数えるか

## この章で答える問い

ここまでの章で、「ルームに今いる人」「途中で抜けた人」「ふたりが同じ瞬間に画面を開いている」といった言葉を何度か使ってきました

これは、サーバの側でルームのメンバーが今いるのかいないのかを正確に追えていることを前提にした表現です

ところが、この「いる、いない」の判定は、見た目より複雑です

ブラウザは閉じられるかもしれないし、タブが切り替わるかもしれません

機内モードに入るかもしれないし、同じ人が 2 タブ開くかもしれません

リロードするかもしれません

どれもサーバ側からは「メッセージが来なくなった」「接続が切れた」としか見えません

この章では、こうした出来事の中で、サーバが「メンバー単位の在席」をどう判定しているかを見ていきます

## ユーザ視点ではどう見えているか

ルームに入ると、メンバー一覧に自分のアバターが並びます

タブを切り替えてしばらくすると、アバターは「focused」から「present」に色が変わります

タブを完全に閉じれば、自分のアバターはルームから消えます

「閉じた直後」ではなく、ほんの少し遅れて消える瞬間があるのに気付くかもしれません

これが grace と呼ばれる猶予の効果で、リロードや一瞬の通信途絶では消さない、という選択をしているためです

## ふたつの単位で数えている

サーバ側で「いる人」を数えるとき、実は 2 つの単位を並行に使い分けています

接続単位とメンバー単位です

### 接続単位

ブラウザがサーバに対して張った 1 本の SSE 接続や WebSocket 接続を、それぞれ別の数として扱います

同じ人が 2 タブ開いていれば、その人だけで接続が 2 本存在することになります

リロードのときは、古い接続が切れて新しい接続が立つので、サーバから見ると一瞬で「－1 してから ＋1」が起きます

接続単位は、ネットワークの実体に近い、ありのままのカウントです

### メンバー単位

「ルームに今いるメンバー」は、接続が何本あろうと、その人ごとに 1 として数えたい単位です

同じ人がリロードして接続が一瞬切れても、メンバーとしてはずっといるまま扱いたいですし、タブを 2 つ開いていても、メンバー一覧には 1 つだけ並んでほしいです

この「人として数える」を実現するために、接続単位の上に、もう 1 段別のロジックを乗せています

## なぜ 2 段に分けるのか

ユーザ目線の「いる」という感覚は、接続の実体とは少しずれています

ブラウザのリロードでは、サーバから見て接続が一旦切れます

そのまま「退出」と判定してしまうと、リロードのたびに他のメンバーの画面で「いなくなりました」と表示され、すぐに「戻ってきました」と再度表示される、という挙動になります

雑談相手にとっては、ただ相手がリロードしただけなのに、画面の上では退室と再入室が走ったように見えます

これを避けるために、接続が切れたあとに少しの猶予を置いて、その間に同じ人の新しい接続が来たら「リロードだったのね」と判定し直す仕組みが必要です

この少しの猶予を grace と呼んでいます

## grace のふたつの長さ

廊下トーク以外の機能では、grace を 2 種類の長さで使い分けています

ひとつは、空気のための 1500 ミリ秒の grace です

リロードや瞬間的な通信途絶を吸収する目的で、短めに設定されています

ある人の SSE 接続が切れた瞬間に「いなくなりました」を配信すると他のメンバーの画面が点滅するため、1.5 秒ほど待って、その間に同じ人の SSE 接続が立ち直ったら退室扱いを取り消します

もうひとつは、ルームの寿命のための 30 秒の grace です

ルームから最後の人が抜けた瞬間にルームを消してしまうと、リロードや瞬間的な通信途絶でルームが消失してしまいます

そこで、最後の接続が切れてから 30 秒待って、その間に誰かの接続が戻ってこなければ初めてルームを片付けます

機能ごとに目的が違うので、grace の長さも別の値で持たれています

## 退出シグナルは 3 経路で受ける

接続が切れる以外にも、ブラウザがやり方を変えて「もういません」を伝えることがあります

サーバは 3 つの経路で退出のサインを受け取ります

### SSE と WebSocket の close

最も普通の経路は、ブラウザの接続が切れることです

SSE では `response` の `close` イベントが、WebSocket では `close` ハンドラがそれぞれ走り、サーバはそこから接続単位の片付けに入ります

普段の使い方ではこれで足りますが、いくつかのケースでは発火しません

### pagehide と sendBeacon

iOS Safari でタブをスワイプアウトする操作では、ブラウザが SSE や WebSocket の close を発火しないまま JavaScript が止まることがあります

このままだとサーバ側にメンバーがずっと「いる」ままの幽霊状態が残り、他のメンバーから見るとアバターがいつまでも消えなくなります

そこで、ブラウザに `pagehide` イベントが届いた瞬間に `navigator.sendBeacon` を使って `POST /rooms/{roomId}/leave` を投げる経路を用意しています

`sendBeacon` はページ離脱時専用の小さな送信 API で、リクエストを投げ捨てて即座にページを閉じても、サーバまで届くことが保証されます

### localStorage によるタブ間ハートビート

ところが、`pagehide` 経路にはひとつ厄介な事情があります

同じ人が同じルームを 2 タブ以上開いているとき、片方のタブを閉じただけで beacon が飛ぶと、サーバ側で「退出した」と判定され、もう片方のタブの接続まで強制クローズされてしまいます

その結果、片方のタブを閉じただけなのに、残ったタブの画面が瞬間的に「離脱した、戻ってきた」とちらつきます

これを避けるために、`localStorage` を使った同 `roomId` のタブ間ハートビートを動かしています

各タブは 1 秒おきに自分のタイムスタンプを `localStorage` に書き込みます

`pagehide` のときに自分以外のタイムスタンプが直近 3 秒以内に残っていれば、別タブが生きていると判定して beacon を投げません

最終タブのときだけ beacon を投げる、という線引きで、iOS の幽霊問題と複数タブのちらつきを両立させています

## 接続カウントは「最後の 1 本」を見る道具

廊下トークだけは、接続単位の上に、もう一段別のカウントを持っています

廊下トークでは、最後の接続が切れたときに、その人がやりかけていた招待や通話の片付けを走らせる必要があります

「最後の接続」 というのが厄介で、同じ人が 2 タブ開いていれば、片方を閉じただけでは最後ではありません

ここで使われているのが、Redis の `HINCRBY` コマンドが返す「演算後の値」です

接続が立ったら ＋1 して、結果が 1 だった接続を初接続として扱います

接続が切れたら －1 して、結果が 0 になった接続を最後の接続として扱います

最後の判定をその場で確定できるので、複数タブが残っているうちは片付けが走らず、本当に最後のタブが切れたときだけ片付けが走ります

廊下トーク以外の機能では、ここまで厳密な最後判定は要らなかったので、ルーム単位の在室を管理する `RoomPresence` の `attach` と `detach` で済ませています

## 実装はどこから読めるか

### ルーム単位の在室

- `apps/backend/application/room/presence.ts`

`attach` と `detach` で接続を増減し、ルームが無人になった瞬間と再度誰か入った瞬間に、`empty` と `populated` のリスナーを呼びます

driver は in-memory と Redis の 2 種類あり、`STORAGE_DRIVER` 環境変数でどちらに倒すかを切り替えます

- `apps/backend/application/room/lifecycle.ts`

ルームの寿命を管理する場所です

`empty` 通知を受けたら 30 秒の grace を張り、その間に `populated` が来なければルームを実際に片付けます

### 空気の grace

- `apps/backend/application/vibe/presence-grace.ts`

`schedule` と `cancel` のふたつだけを持つシンプルな port 型です

退室検知時に 1500 ミリ秒の猶予を張り、再入室で `cancel` されれば配信を取りやめます

`cancel` の戻り値が `true` のときは、再入室なので新規参加ではなく状態更新として扱う、という判断にも使われます

### 廊下トークの接続カウント

- `apps/backend/application/hallway/connection-counter.ts`

`attach` と `detach` の戻り値で `isFirst` と `isLast` を返します

呼び出し側は `isLast` が `true` のときだけ片付け処理を走らせて、招待や通話の掃除を確実に最終接続に限定します

### ブラウザ側の退出シグナル

- `apps/frontend/views/room/use-leave.ts`

`pagehide` イベントを購読し、`localStorage` 上のハートビートで他タブ生存を確認してから、最終タブのときだけ `navigator.sendBeacon` で leave エンドポイントを叩きます

- `apps/frontend/features/vibe/use-visibility.ts`

`document.visibilityState` をもとに `present` と `focused` の送信を自動化しています

`hidden` 方向は BFCache 入りで JS が凍結される経路を踏むため、デバウンスせず即時送信します

`visible` 方向はタブの往復を 8 秒のトレーリングデバウンスで 1 通信にまとめます

## まとめ

サーバが「ルームにいる人」を正確に追うために、接続単位とメンバー単位の 2 段で数えています

接続単位は、ネットワークの実体に近い、生のカウントです

メンバー単位は、リロードや短い瞬断を grace で吸収して、ユーザの感覚に合う「いる」を作る層です

退出のシグナルは 1 本の経路では足りず、SSE と WebSocket の close、`pagehide` からの `sendBeacon`、`localStorage` のタブ間ハートビート、この 3 つを組み合わせて取りこぼしを減らしています

廊下トークだけは、招待や通話の片付けを最後の接続に限定するため、接続カウントの戻り値で初接続と最終接続を判定する仕組みを別に持っています

第 7 章では、ここまで暗黙にサーバ 1 台として話してきた構成を、複数台に増やしたときに何が変わるかを見ていきます
