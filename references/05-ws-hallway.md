# 第 5 章 WebSocket で動く廊下トーク

## この章で答える問い

リアルタイム通信の 4 機能のうち、廊下トークだけが WebSocket を選んでいます

「双方向のやり取りが要る」という抽象的な説明では、なぜ片方向の SSE で足りないのかが具体的に見えません

この章では、廊下トークで動く「招待」と「通話」のふたつの状態を取り上げて、それぞれが両側からほぼ対等にメッセージを飛ばし合う様子を見ていきます

## 廊下トークでユーザに何が起きているか

ルームの右側にはメンバー一覧が並んでいます

誰かのアバターを押して「話しかけたい」を選ぶと、相手の画面に通知が届きます

相手が「応じる」を押すと、ふたりだけの 1 対 1 の通話が始まります

通話中は、テキストメッセージのやり取りができ、ミュートを切り替えたり、どちらかが「終わる」を押せば通話が終わります

招待を取り消す、相手の招待を断る、通話を切る —— こうした操作はどちらの側からも起こせて、起こした瞬間に相手の画面に状態が伝わります

## 招待と通話で動くふたつの状態

廊下トークでは、ふたつの状態が時間軸の上で順番に動きます

ひとつめは「招待」です

「話しかけたい」を押した瞬間に始まり、相手が応じるか断るか、送り手が取り消すまでの短い窓です

ふたつめは「通話」です

招待が応じられた瞬間に始まり、どちらかが切るまで続きます

```
招待を出す ──── 招待中 ──── 応じる ──── 通話中 ──── 切る ──── 終了
                  │
                  └─ 取り消す or 断る ─→ 終了
```

招待が短い「始まりかけ」の窓だとすれば、通話はそのあとずっと続く長い時間です

ふたつの状態は性質が違いますが、同じ通信路の上で切れ目なく扱う必要があります

## なぜ片方向の SSE では足りないか

招待でも通話でも、両側からほぼ対等な頻度でメッセージが飛びます

招待を例に取ると、片方が出した瞬間に相手の画面に通知が届く必要があり、相手が応じる返事も逆向きに即座に届く必要があります

通話に入ると、テキストメッセージはそれ以上に対称です

```
ブラウザ A ── 「メッセージを送ります」 ── ブラウザ B
ブラウザ A ←─ 「メッセージが届きました」 ── ブラウザ B
ブラウザ A ── 「ミュートにしました」  ── ブラウザ B
ブラウザ A ←─ 「相手がミュートを解除しました」 ── ブラウザ B
```

これを SSE と HTTP の組み合わせで組もうとすると、サーバからブラウザへの向きは SSE で受け取り、ブラウザからサーバへの向きは HTTP リクエストごとに張り直すことになります

メッセージごとに新規 HTTP 接続を立てる動きは、雑談のテンポにはまず合いません

加えて、招待や通話では「相手はまだいるか」を即座に追いかける必要があります

WebSocket であれば、接続が落ちた時点で即「切れた」と判定できます

SSE の片方向では、サーバ側はブラウザの離脱を検知できる一方で、ブラウザ側からサーバ側の状態は感じ取りにくくなります

両方向に同じだけ流せて、接続が張りっぱなしで切れたかどうかが分かる WebSocket が、双方向の対話と切断検知の両方をシンプルに収めてくれます

## ネットワークでは何が流れているか

WebSocket の接続は、最初の HTTP リクエストの中で「これから WebSocket に切り替えます」と宣言する Upgrade のやり取りから始まります

```
ブラウザ ── GET /rooms/{roomId}/hallway ─── サーバ
              Upgrade: websocket
ブラウザ ←─ 101 Switching Protocols ─────── サーバ
```

ここで HTTP のセッションが終わり、同じ TCP 接続が両方向のメッセージ通路に変わります

そのあとは、サーバとブラウザの両方が、いつでもメッセージを送れます

```
ブラウザ ←─ Ping              ── サーバ
ブラウザ ── Pong              ── サーバ
ブラウザ ── InviteMember      ── サーバ
ブラウザ ←─ MemberInvited     ── サーバ
ブラウザ ←─ InvitationAccepted── サーバ
ブラウザ ── SendMessage       ── サーバ
ブラウザ ←─ MessageReceived   ── サーバ
```

メッセージは、`name` と `data` のふたつのフィールドで揃えた封筒に詰めて送ります

`name` でメッセージの種類を示し、`data` でその中身を運ぶ仕組みです

## SSE のときとは違う 4 つの手当て

WebSocket 接続を実用に耐える形にするには、SSE と同じく裏側で必要になる仕組みがあります

### Ping と Pong による生存確認

サーバは 20 秒おきに `Ping` メッセージをブラウザに送ります

ブラウザは即座に `Pong` を返します

`Pong` が一定時間返ってこなかった接続は、ゾンビ接続と見なしてサーバ側から閉じます

SSE の heartbeat と似ていますが、こちらは双方向のやり取りで生存を確認する点が違います

### close code

WebSocket は、接続を閉じるときに「なぜ閉じたか」を表す番号 —— close code —— を相手に送れます

`1000` は正常終了、`1001` はブラウザのタブが閉じられたことを示し、`4001` から `4004` はプロダクト固有の理由を割り当てています

たとえば、不正なフレームを受け取ったら `4001` で閉じ、認可に失敗したら `4002` で閉じる、といった使い分けがされます

### Upgrade 時の認可

WebSocket は最初の HTTP リクエストで切り替わるので、認可の判定はその HTTP 段階でやる必要があります

廊下トークでは、URL のパターン、Origin ヘッダ、Cookie の 3 つを Upgrade のタイミングで照合し、通った接続だけが WebSocket に切り替わります

接続が立ったあとの判定は遅すぎて、不正な接続のメッセージが流れ始めてしまいます

### 接続単位での後始末

WebSocket の接続が切れたら、その人がやりかけていた招待や通話の状態をそろえて片付ける必要があります

招待中だったら相手側に「取り消されました」と伝え、通話中だったら相手側に「切れました」と伝えなければ、相手の画面が止まったままになります

接続切れ、状態の整理、相手への通知、この 3 段階を毎回確実に走らせるのが、廊下トークの実装の半分くらいを占めています

## 実装はどこから読めるか

### プロトコル定数

- `packages/transport-protocol/ws.ts`

Ping の送出間隔 20 秒、Pong の許容超過 10 秒、close code の辞書が並んでいます

close code には、RFC 6455 の標準コード `1000`、`1001`、`1008` と、プロダクト固有コード `4001` から `4004` の両方が定義されています

### WebSocket プロトコル本体

- `apps/backend/infrastructure/transport/ws/connection.ts`

`name` と `data` の 2 段封筒の組み立て、JSON 化、close の冪等化を担います

NestJS 公式の Socket.io アダプタを使わず、`ws` ライブラリの生 WebSocket を直接ラップしているのが学習目的としての骨子です

- `apps/backend/infrastructure/transport/ws/heartbeat.ts`

20 秒ごとに `Ping` を送り、`Pong` の到着時刻を記録、超過したら `PongTimeout` で閉じる、というシンプルな実装です

- `apps/backend/infrastructure/transport/ws/hub.ts`

接続単位の購読管理、ルームトピックへの参加、`pagehide` 起点の能動 close を集約します

### 廊下トークのエンドポイント

- `apps/backend/presentation/ws/hallway.gateway.ts`

HTTP サーバの `upgrade` を横取りして、URL パターン、Origin、Cookie の三段認可を行い、通過したら `WsConnection` を組み立てて hub に預けます

- `apps/backend/presentation/ws/hallway-dispatch.ts`

ブラウザから届いた `name` を見て、対応する usecase に振り分ける薄いハブです

`InviteMember`、`AcceptInvitation`、`DeclineInvitation`、`CancelInvitation`、`SendMessage`、`LeaveCall` といった命令名と usecase を結び付けています

### ブラウザ側の WebSocket クライアント

- `apps/frontend/libraries/transport/ws/native.ts`

ブラウザの組み込み `WebSocket` を 1 段ラップして、Zod でメッセージごとのペイロードを検証してからアプリ側のハンドラに渡しています

`Ping` を受けたら自動で `Pong` を返す処理も、ここに閉じ込められています

- `apps/frontend/libraries/transport/ws/resilient.ts`

通信が途切れたときに自動で再接続する仕組みがここにあります

ネットワークの瞬断のたびにユーザに再操作させなくて済むようになっています

## まとめ

廊下トークが WebSocket を選んだ理由は、招待と通話のどちらでも、両側からほぼ対等な頻度でメッセージが飛び交うためです

その双方向の対称性を SSE と HTTP の組み合わせで真似しようとすると、メッセージごとに HTTP 接続を立てる動きや、切断検知の難しさで、雑談のテンポが壊れてしまいます

WebSocket は「同じ電話線で両方からしゃべる」という生地に、Ping と Pong の生存確認、close code の理由付き終了、Upgrade 時の認可、接続切れ後の片付けを編み込んで成り立っています

第 6 章では、ここまで何度か言葉を出してきた「ルームに今いる人」をサーバはどう数えているのか、タブが閉じられたことをどう知るのか、こうした「いる、いない」の判定にまわる仕組みを見ていきます
