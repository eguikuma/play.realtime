# 第 4 章 SSE で動く 3 機能

## この章で答える問い

第 3 章で、空気、作業音、ひとこと の 3 機能が SSE で動いていると並べました

ここからは、その 3 機能が画面でどう動いているかを起点に、ネットワークでは何が流れているのか、リポジトリのどこを読めばその実装に出会えるのかを順に降りていきます

「サーバが一方向に流すだけ」と説明したものの、実物の SSE には heartbeat や retry や close 検知といった生地の手当てが必要です

それも合わせて見ていきます

## 3 機能はユーザの目にどう映るか

### 空気

ルームに入ると、参加者ぶんのアバターが画面に並びます

誰かが少しでもキーボードやマウスを動かしていれば、そのアバターには「focused」というラベルが付いて、集中している様子を表す色で表示されます

何もせずに少し経つと、そのアバターは「present」というラベルに戻って、画面を開いているだけの色になります

自分は画面を眺めているだけでも、隣の人のアバターの色がふわっと変わっていきます

### 作業音

誰かが BGM ボタンを押して曲を選ぶと、その曲名と再生ボタンがルーム全体に共有されます

別の誰かが曲を切り替えれば、自分の画面の表示も即座にそちらに切り替わります

ルームのメンバー全員が、同じ曲のリストを同じ瞬間に見ている状態になります

### ひとこと

下のフォームに短いメッセージを書いて送ると、自分の画面と他のメンバーの画面の両方に、同じ瞬間にメッセージが浮かびます

過去の投稿は、入室直後に「これまでの流れ」として一括で届きます

途中から入った人も、置いていかれずに今までの流れを追えます

## ネットワークでは何が流れているか

3 機能とも、サーバに対して 1 本の長い HTTP レスポンスを開きっぱなしにすることで成立しています

```
ブラウザ ── GET /rooms/{roomId}/murmurs/stream ── サーバ
ブラウザ ←─ event: snapshot                     ── サーバ
ブラウザ ←─ event: posted                       ── サーバ
ブラウザ ←─ : heartbeat                         ── サーバ
ブラウザ ←─ event: posted                       ── サーバ
```

最初のリクエストに対して 200 が返ってきますが、レスポンスはそこで終わりません

サーバは `Content-Type: text/event-stream` を宣言して、レスポンスボディに 1 件ずつイベントを書き足し続けます

ブラウザは `EventSource` という組み込みの仕組みでこのストリームを受け取り、イベント名ごとに別々の処理に振り分けます

## 実は流れている 3 つのもの

「サーバが流すだけ」と言うとシンプルですが、実物の SSE にはユーザイベントの裏側で必要になる仕組みが 3 つあります

### heartbeat

サーバは 15 秒ごとに `: heartbeat` というコメント行を流しています

これは画面に表示するイベントではなく、接続が生きているかどうかを中継のプロキシやロードバランサに気付かせるための合図です

途中の経路はアイドル接続をタイムアウトで切ってくることがあるので、定期的に何かを書いてやる必要があります

### retry

接続が切れたとき、ブラウザの `EventSource` は自動で再接続を試みます

サーバは最初に `retry: 3000` という行を流して、「3 秒後にもう一度繋ぎに来てね」と再接続待ち時間を指定します

これによって、ネットワークが瞬断しても数秒後に黙ってつながり直します

### close 検知

ユーザがタブを閉じた、あるいはネットワークが落ちたといった理由でブラウザ側の接続が終わると、サーバ側にもその情報が `response` の `close` イベントとして届きます

サーバはこれを購読解除と heartbeat タイマーの停止のきっかけにします

これがないと、もういない接続に向けて延々と書き続ける状態になり、メモリやプロキシの接続枠を浪費します

## 実装はどこから読めるか

リポジトリの中で、SSE まわりの実装は 3 つの層に分かれています

### プロトコル定数

- `packages/transport-protocol/sse.ts`

heartbeat の間隔 15 秒、retry の待ち時間 3 秒、`Content-Type` の文字列、共通イベント名として `snapshot`、`update`、`heartbeat` の 3 つが並んでいます

短いファイルなので、最初に開いて全体の値感を掴むと、他のファイルが読みやすくなります

### SSE プロトコル本体

- `apps/backend/infrastructure/transport/sse/connection.ts`

ヘッダの送出、`retry` フィールドの送信、`event` 名つきの 1 件配信、コメント行の送出、close の冪等化、ここで全部やっています

NestJS の `@Sse()` デコレータをあえて使わず、Express の `Response` に直接書き込んでいる —— というのが学習目的としての筋です

- `apps/backend/infrastructure/transport/sse/heartbeat.ts`

15 秒ごとに `: heartbeat` コメントを送るだけのシンプルな実装です

- `apps/backend/infrastructure/transport/sse/hub.ts`

接続単位の購読管理と、PubSub から流れてくるイベントを「どの接続に配るか」に引き合わせる真ん中の層です

### 機能ごとの SSE エンドポイント

- `apps/backend/presentation/http/vibes.controller.ts`
- `apps/backend/presentation/http/bgms.controller.ts`
- `apps/backend/presentation/http/murmurs.controller.ts`

それぞれが `GET /rooms/{roomId}/{機能}/stream` を持っていて、SSE 接続を 1 本開き、対応するトピックに購読を載せます

ひとことの controller には、購読が始まった瞬間に過去の投稿一覧を `Snapshot` イベントとして配り直す挙動も入っています

途中から接続した人にも、過去の流れが見える仕組みです

### ブラウザ側の購読

- `apps/frontend/libraries/transport/sse/native.ts`

ブラウザの `EventSource` を 1 段ラップして、Zod でイベントごとのペイロードを検証してから、アプリ側のハンドラに渡しています

「学習として隠蔽しないけれど、検証だけは挟む」という線引きが、このファイルに集約されています

## まとめ

3 機能のいずれも、サーバから 1 本の長い HTTP レスポンスを開きっぱなしにして、その上にイベントを書き続けるという同じ生地で動いています

その生地には heartbeat と retry と close 検知が編み込まれていて、実用に耐えるためにはこの 3 つのどれも欠かせません

第 5 章では、この生地で足りなくなる場面 —— 廊下トーク —— に踏み込んで、なぜ WebSocket でないと組めないかを見ていきます
