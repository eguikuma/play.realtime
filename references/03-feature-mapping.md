# 第 3 章 りもどきの使い分け

## この章で答える問い

第 2 章でリアルタイム通信のやり方が 3 つあると見ました

ここからは、りもどきの中でそれぞれをどう使い分けているかに踏み込みます

機能ごとに違う通信方式を選んでいる理由を言語化することが、この章の目的です

一覧を頭に入れておくと、第 4 章以降の各論がひとつの判断軸に繋がって読めます

## 4 つの機能と通信方式の対応

りもどきには 4 つの機能があります

それぞれがどの通信方式で動いているかを並べると、こうなります

| 機能 | 何ができるか | 使っている通信 |
| --- | --- | --- |
| 空気 | メンバーの集中度がアバターで見える | SSE |
| 作業音 | 誰かが流している BGM が共有される | SSE |
| ひとこと | 投稿された短いメッセージが画面に流れる | SSE |
| 廊下トーク | ふたりで雑談する 1 対 1 の通話 | WebSocket |

4 機能のうち 3 機能は SSE で動いています

廊下トークだけが WebSocket になっていて、このアンバランスさには理由があります

## サーバから流すだけで足りるか

判断軸はひとつだけです

サーバから一方向に送るだけで足りるか、という問いです

空気、作業音、ひとこと の 3 つは、どれも「サーバ側で何かが起きたら全員に届けたい」という形をしています

誰かが集中モードに入った、誰かが BGM を変えた、誰かが「ひとこと」を投稿した —— こうした出来事は、サーバから各クライアントに流すだけで完結します

クライアントから何かをサーバに送りたいときは、別途 HTTP リクエストを投げれば良いので、双方向の通信路をわざわざ張る必要はありません

片方向で済む場面では SSE が向いています

## 廊下トークだけ性質が違う

廊下トークは違う性質を持っています

ふたりが同じ瞬間に画面を開いていて、招待を出し、相手が応答し、通話が始まり、どちらかが切るまで状態が変わり続けます

この最中に、招待を取り消す、通話を切る、ミュートを切り替えるといった操作が、両側からほぼ対等な頻度で飛び交います

やり取りを HTTP リクエストで張り直していたら、遅延が積み重なって雑談として成立しなくなります

両方向に同じだけ流せて、接続が張りっぱなしになる WebSocket が向くのは、こういう場面です

## なぜ全部 WebSocket にしなかったか

学習目的のプロダクトとして、あえて使い分けを残しています

空気や作業音を WebSocket で実装すること自体は技術的に可能です

ただ、双方向が要らないものまで WebSocket にすると、接続管理が複雑になり、クライアントから不要なメッセージを受け付ける入り口を増やすことになります

単純な押し出しで足りる機能は、SSE のほうが書く量が少なく、壊れにくくなります

「双方向の通信が必要か」を判断軸にして、必要なときだけ WebSocket を選ぶ —— この使い分けの感覚を身につけることが、第 4 章以降の各論を読む土台になります

## 実物はどこにあるか

リポジトリの中で、この使い分けが見えるのは主に次のファイルです

- `apps/backend/presentation/sse/` —— SSE で配信している 3 機能の入り口
- `apps/backend/presentation/ws/` —— WebSocket で配信している廊下トークの入り口
- `packages/transport-protocol/` —— 両方の通信仕様が並んで定義されている

次章では SSE 側に絞って、中で何が流れているかを見ていきます
